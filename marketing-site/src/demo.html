AI Mechanics — How Each Suggestion Works
◉  Employee Portal
Role SuggestionsLLM + RAG
How it’s computed

Employee’s skills, seniority, and specialisation are serialised into a prompt. The active AI Workflow provider (Claude / Perplexity / OpenAI / Gemini / self-hosted) is called via get_role_suggestions_for_profile().

RAG searches ai_documents for relevant roles-catalogue and trend chunks, prepends them as context, then asks the LLM to return the top N most relevant roles for this profile.

A deterministic match_score (0.0–1.0) is appended per role: |emp_skills ∩ required| / |required|. Roles below min_match_score are filtered before returning.

// employee → role_suggestions { "enabled": true, "ttl_hours": 24, "model_override": null, "guardrails": { "max_roles": 5, "min_match_score": 0.3 } }

Learning PathLLM + RAG
How it’s computed

Skill set, career goal, and seniority gap are combined with RAG context (skill docs, cert data). The LLM is asked to return a node-graph JSON with node types: skill, cert, project, milestone.

Output is validated for JSON structure. Nodes exceeding max_nodes are trimmed (lowest-priority last). If include_certs: false, cert nodes are stripped. Cached 48h — long enough to avoid redundant calls, short enough to reflect market shifts.

Opportunity Match ScoreDeterministic
How it’s computed

No LLM. For every published opportunity, at query time:

overlap = emp_skills ∩ opp.required_skills score = len(overlap) / len(required_skills) # if required_skills empty → 0.0

Badge colour: green ≥70%, yellow 40–69%, grey <40%. Score attached to every row in the opportunities list — no separate call needed.

Guardrails (coverage_config)
// employee → opportunities_matching { "enabled": true, "ttl_hours": 6, "model_override": null, "guardrails": { "min_match_score": 0.2 } }

Opportunities below min_match_score are still shown but de-emphasised. Disabling the feature hides all badges.

Audit log entry

No row written. The score is computed inline per opportunity in the same SQL query that returns the list — it is not a discrete suggestion. The coach’s “Suggest Employees for Opportunity” uses the same scoring formula and does write an audit row (see Coach Portal → Suggest Employees).

// computed inline; nothing in ai_suggestion_logs GET /api/v1/opportunities → [{ id, title, type, required_skills, match_score: 0.67, overlap: ["Python", "SQL"] }, …]
Career Advice (Get Advice)LLM + 4 pillars
How it’s computed

Single-shot question on the dashboard’s Get Advice card. Backend assembles a strict 4-pillar context: anonymised profile (skills resolved from SkillTaxonomy — UUIDs never sent), top 5 active opportunities ranked by skill overlap, the latest Industry 4.0 readiness snapshot, and the active workflow’s company strategy notes.

Profile too thin (< 3 skills, no specialisation/seniority/career goal) → 422 refusal rather than fabrication. Off-context questions (e.g. “where did I go to university?”) get a clean “not in your profile, please add it” refusal.

Output passes a regex scrub for [1][2] / [digest] citation markers and stray UUIDs. Rendered as Markdown via the shared CfMdText component (marked + DOMPurify).

Suggest Employees for OpportunityDeterministic
How it’s computed

Coach asks “who in the org should I shortlist for this opportunity?”. No LLM. Pool: all employees meeting the opportunity’s minimum seniority. Score per candidate:

skill_score = |emp_skills ∩ opp.required_skills| / |required| seniority_score = max(0, 1 - |emp_idx - min_idx| * 0.15) availability_bonus = 0.1 if emp.available_for_mentoring else 0 match_score = skill_score * 0.65 + seniority_score * 0.25 + availability_bonus

Top-N returned (default 5), with skill overlap and mentoring-availability flag.

Guardrails (coverage_config)
// coach → opportunities_ai_match { "enabled": true, "ttl_hours": 2, "model_override": null, "guardrails": { "top_n": 5, "min_match_score": 0.3 } }

Zero LLM cost. Tagged model_used: "rule_based" so the audit page can distinguish it from LLM rankings.

Audit log entry
{ "suggestion_type": "opportunity_match", "portal": "coach", "feature": "opportunities_ai_match", "employee_pseudo_id": null, // opp-scoped "model_used": "rule_based", "payload_json": { "opportunity_id": "…", "matches": [ { "pseudo_id": "EMP-A", "match_score": 0.92 }, … ] } }
Mentor Match (employee & opportunity)Deterministic
How it’s computed

Two sibling endpoints share one scoring formula:

  • GET /employees/{pseudo}/mentor-suggestions — on the employee detail page
  • GET /opportunities/{id}/mentor-suggestions — on the opportunity detail page

Pool: employees with available_for_mentoring=true, seniority ≥ target’s, pseudo_id ≠ target. Weighted score per candidate:

skill_score = |mentor_skills ∩ target_skills| / |target_skills| sen_gap = seniority_rank(mentor) - seniority_rank(target) seniority_score = max(0, 1 - max(0, sen_gap - 2) * 0.15) load_score = 1 - (active_assignments / 10) // from mentor_assignments table match_score = skill_score * 0.55 + seniority_score * 0.30 + load_score * 0.15
Guardrails (coverage_config)
// coach → opportunities_ai_match { "enabled": true, "ttl_hours": 24, "model_override": null, "guardrails": { "min_match_score": 0.2, "max_mentors": 5 } }

Candidates below min_match_score excluded. Capped at max_mentors. Zero LLM cost — entirely data-model driven. Only pseudo-ids exposed — no names. Tagged model_used: "rule_based" in the audit log.

Opportunity AI Candidate MatchLLM + Deterministic
How it’s computed

Phase 1 — Deterministic (all 212 employees):

for emp in all_employees: overlap = emp.skills ∩ opp.required_skills score = len(overlap) / len(required) candidates.append({ emp, score, overlap, missing })

Sorted by score desc. Filtered by min_match_score. Capped at max_candidates.

Phase 2 — LLM narrative (top 3 only): opportunity description + top-3 anonymised profiles sent to the active provider for a concise hiring narrative per candidate.

Short 2h TTL — opportunity requirements may change. LLM only sees: pseudo_id, specialisation, seniority, skill_list. No names, no email, no personal data.

⊕  AI Workflow Connector — Audit & Config
Outcome Feedback LoopDid it work?
How it works

The audit log answers “what did the AI say?”. The new ai_suggestion_feedback table answers “did it work?”. Each suggestion log row can be paired with one or more feedback events:

  • accepted — user actioned the suggestion (applied to a role, enrolled in a learning node, asked a follow-up)
  • dismissed — user explicitly rejected it
  • rated — thumbs / 1–5 stars; carries a numeric rating
  • auto — system-observed outcome (readiness delta 30d after a learning path was generated, application result, etc.)

Joining the two over time gives the platform real signal: acceptance rate per suggestion_type, average rating, correlation with downstream readiness changes — the data needed to tune prompts on evidence rather than vibes.

Endpoints
// Employee — record outcome on one of *their* suggestions POST /api/v1/ai-suggestions/{log_id}/feedback { "action": "accepted" | "dismissed" | "rated", "rating": 1..5, // required when "rated" "note": "free text", "payload": { /* action-specific */ } } // Employee — their own feedback history (self-review) GET /api/v1/ai-suggestions/my-feedback // Admin — rollup per suggestion_type GET /api/v1/admin/ai-workflow/audit/feedback-summary → [{ suggestion_type, total_suggestions, feedback_events, feedback_rate, accepted, dismissed, rated, avg_rating, acceptance_rate }, …] // Admin — suggestion detail now includes feedback events array GET /api/v1/admin/ai-workflow/audit/suggestions/{log_id} → { …, feedback: [{ action, rating, recorded_at, … }] }

Authorisation on POST: the suggestion must either belong to the caller (matching employee_pseudo_id) or be portal-wide / unscoped. Bad action / out-of-range rating → 422.

Telemetry
// Prometheus counter, labelled per outcome cf_suggestion_calls_total{ suggestion_type="coaching_advice", portal="employee", outcome="feedback_accepted" } // Sample summary endpoint output { "suggestion_type": "coaching_advice", "total_suggestions": 16, "feedback_events": 2, "accepted": 1, "dismissed": 0, "rated": 1, "avg_rating": 4.0, "acceptance_rate": 1.0 }
AI Observability — Prometheus metricsGrafana-ready
What is captured

The HTTP-level instrumentor already records request latency and status. These metrics target the AI surface — the things you can’t infer from request logs:

  • cf_llm_calls_total{provider, model, outcome} — success / empty / http_error / exception
  • cf_llm_call_duration_seconds{provider, model} — histogram, 0.5..90s buckets
  • cf_suggestion_calls_total{suggestion_type, portal, outcome} — success / empty_answer / refused_thin_profile / feedback_accepted / feedback_dismissed / feedback_rated
  • cf_llm_artifacts_scrubbed_total{kind} — citation_numeric / citation_named / uuid
  • cf_ai_cache_lookups_total{cache_kind, outcome} — hit / miss / expired
Why it matters

A regression like the Perplexity citation leak we caught manually mid-test would now alert itself: cf_llm_artifacts_scrubbed_total{kind="citation_numeric"} stops being zero. Same for silent regressions:

  • P95 latency for one provider creeping past 30s → switch model_override
  • refused_thin_profile ratio rising → onboarding wizard issue
  • empty_answer spikes on one suggestion_type → bad prompt template change
  • Cache hit rate dropping → TTL too aggressive or fingerprint collision

Cost & Abuse ControlsThree layers
Three independent gates
  • Per-employee daily quota — default 50 LLM-backed calls per pseudo_id per UTC day. /ai-advice and /ai-chat consume one slot each. Exhausted → HTTP 429 with a clear message; resets at 00:00 UTC. Redis-backed (INCR + EXPIRE); fail-open if Redis is down.
  • Per-provider circuit breaker — after 5 consecutive failures (HTTP error / exception) the breaker opens for 120s. Open-state short-circuits the LLM call and returns ""; caller serves the configured fallback. Auto-closes after the cooldown.
  • Monthly token cost cap — rough estimate (chars/4) recorded per provider in cf:cost:YYYY-MM. coverage_config.__cost__.monthly_token_cap on the active workflow sets the limit; once hit, every LLM call returns "" until the next month. Unset = unlimited.

All three are fail-open: if Redis or a counter is unreadable, the call proceeds rather than blocking the portal.

Audit Evidence Trail — verbatim prompt & answerCompliance-ready
What is captured

Every conversational suggestion (coaching_advice, ai_chat) records into payload_json:

  • The system prompt verbatim (the strict-grounding rules + role declaration)
  • The full user prompt — all four PROFILE / OPPORTUNITIES / READINESS / STRATEGY blocks exactly as the model saw them
  • The full answer after post-LLM scrub
  • The context_used breakdown (skills count, opportunities ranked, readiness/strategy presence)
  • Every recorded ai_suggestion_feedback event (accepted / dismissed / rated 1–5)

This is the artefact HR / compliance / legal asks for when reviewing why the AI made a recommendation. No PII is in any of these blocks — only pseudo-IDs and aggregated profile fields, by construction.

Why it matters

Three audiences, one artefact:

  • Compliance / GDPR Article 22 — demonstrates that any AI-assisted decision had a documented basis (the four pillars) and that the user retained the ability to contest / accept / dismiss.
  • Coach / HR review — when an employee disputes advice, the conversation is fully reproducible.
  • Prompt engineering — reading the actual prompt the model saw is the fastest way to debug a regression.
Runtime FE Configuration — one bundle deploys everywhereDevOps
How it works

The employee portal reads its environment-specific values (Keycloak URL, realm, client-id, API base URL) from window.__APP_CONFIG__ at runtime, populated by a /config.js file rendered from container env at startup.

Build-time VITE_* values still work as a fallback so pnpm dev and existing built bundles keep working unchanged.

Same image promotes from demo → staging → production. Set CF_KEYCLOAK_URL, CF_EMPLOYEE_API_URL, … on the deployment and they take effect on next pod restart.

Audit Trail — every suggestion, every portalAppend-only
How it works

Every call to any AI function invokes _log_suggestion(portal=, feature=) before returning. The TTL is resolved by _get_feature_ttl(workflow, portal, feature): reads coverage_config[portal][feature].ttl_hours from the active config, falls back to suggestion_ttl_hours global, then to 24h hardcoded default.

The log is append-only — no UPDATE or DELETE is ever issued. Admins browse it via Admin → Connectors → Suggestion Audit tab, filterable by portal, feature, pseudo-id, or date range.

Five-layer cost & resilience stackall providers
What it does

Sits in front of every conversational AI endpoint. Layers 1–4 ship benefits regardless of which provider the active workflow uses; layer 5 sits behind a capability predicate that activates only when Anthropic is selected.

  1. Prompt-shape budget — top-3 opportunities, strategy 800 chars, chat history 4 turns. 25–35% input-token cut on every provider.
  2. Response-hash cache — Redis key cf:airesp:<sha>, 5-min TTL. Double-clicks & reloads cost zero.
  3. Multi-replica circuit breaker — Redis-backed open state across all replicas.
  4. Connector freshness gauge — cf_connector_last_sync_seconds + alert at 24h.
  5. Anthropic prompt cache — ephemeral cache_control on the system prompt; ~90% input-cost cut on cache hits.
Cache key (layer 2)
# services/ai_connector/cache.py sha256( pseudo_id | question.strip().lower() | workflow_id | skill_count | ",".join(sorted(opp_top_ids)) | ...extra ) → "cf:airesp:<hex>"

/ai-chat additionally folds a hash of recent turns in so a follow-up question with the same words can’t collide with the original.

Observability
# Prometheus metrics emitted cf_suggestion_calls_total{outcome="cache_hit"} cf_prompt_cache_tokens_total{provider,direction} cf_connector_last_sync_seconds{connector} cf_llm_calls_total{provider,model,outcome} cf_llm_artifacts_scrubbed_total{kind}

Fail-open everywhere: cache, breaker, and quota helpers all degrade to no-op when Redis is unreachable.

Vector cosine RAG retrievalall providers
What it does

Every AI function that needs grounding chunks from ai_documents calls _cached_text_search(). That helper now embeds the query via the configured embedding provider (Ollama nomic-embed-text by default, OpenAI text-embedding-3-large when OPENAI_API_KEY is set) and orders ai_document_vectors by cosine distance.

Cosine similarity matches synonyms and paraphrases that LIKE missed: "AWS" ↔ "Amazon Web Services", "k8s" ↔ "Kubernetes", "microservices" ↔ "service-oriented architecture".

SQL shape
# services/ai_connector/rag.py SELECT d.title, d.source_name, substring(v.chunk_text from 1 for 600), 1 - (v.embedding <=> CAST(:qvec AS vector)) AS score FROM ai_document_vectors v JOIN ai_documents d ON d.id = v.document_id WHERE :ns = '' OR d.source_name = :ns ORDER BY v.embedding <=> CAST(:qvec AS vector) LIMIT :k
Fallback

If embedding generation fails (no embedding service reachable, transient HTTP error) or the vector query returns nothing, falls through to the original LIKE-based search. Both paths return the same dict shape, so every existing caller is unchanged. RAG is never blocked.

Daily AI-quota indicatorUX
What it does

The employee portal sidebar footer shows AI: 48 / 50 today under the user name, polled every 60s. Turns orange when remaining ≤ 5 so the user gets a visible heads-up before hitting the daily ceiling. Hidden when Redis is unreachable (remaining = -1) so the chip never lies.

Reads from GET /api/v1/quota/me which returns the same Redis counter the Phase 1 daily-quota breaker enforces — single source of truth, no drift between what the user sees and what the platform actually applies.

Endpoint shape
# services/employee/routers/quota.py GET /api/v1/quota/me → { "remaining": 48, "limit": 50, "resets_at": "2026-04-30T00:00:00+00:00" }

Empty remaining = -1 when the Redis sidecar is down — same fail-open posture as the rest of Phase 1.

Where it surfaces

For now, employee portal only (apps/employee/src/layouts/AppLayout.vue). Coach / manager / admin portals don’t have a quota concept on their AI surfaces yet, so no chip there. The endpoint itself is auth-scoped to the calling Keycloak subject so each employee only sees their own counter.

Drift gate (two-gate setup)CI
What it catches

Two complementary CI gates, paired so neither bug class slips through:

  • alembic check — fails when a SQLModel field has no matching migration column. The bug class that manifests as UndefinedColumn 500s after a fresh deploy.
  • pytest test_raw_sql_ghost_tables.py — fails when a router uses raw SQL against a table that has no SQLModel at all. The bug class behind the api_keys incident.
How alembic check stays useful

migrations/env.py ships an _include_object filter that skips index, FK, and unique-constraint naming divergence (cosmetic only — same b-tree, same referential semantics). Real drift — missing columns, nullable changes, missing tables — is still caught.

# migrations/env.py def _include_object(obj, name, type_, ...): if type_ in {"index", "foreign_key_constraint", "unique_constraint"}: return False # skip cosmetic return True
First catches

api_keys table: admin /api-keys returned 500 because the table referenced via raw SQL had no SQLModel and no migration. Caught by the ghost-table test.

coach_forms + skill_taxonomy: SQLModel had fields (employee_committed_at, cc_signed_off_at, manager_signed_off_at, opportunity_id, description) the migrations never created. Caught by alembic check when the strict gate was restored; reconciled via migrations 20260429_0002 + _0003 (both idempotent).
