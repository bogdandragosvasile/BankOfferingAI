AI in CareerForge

A complete technical breakdown of how artificial intelligence is embedded in CareerForge — architecture, implementation code, guardrails, current strengths and weaknesses, and the path toward a self-improving, simulation-driven platform.

Last updated: 23 April 2026  ·  Internal use only

What AI Does in CareerForge

AI in CareerForge is not a single feature — it is an intelligence layer that runs across all four portals, generating personalised career guidance, synthesising market data, and building structured learning paths. Every AI call is grounded in real business context (the employee's profile, skill inventory, anonymised coaching relationship) and constrained to JSON-structured output.

🗺️
Learning Paths
Claude generates a personalised node-graph learning path for each employee at registration, spanning skills, certifications, projects, and milestones.
🎯
Role Suggestions
Based on the employee's specialisation and seniority, AI proposes 5 emerging roles with match scores and required skills.
📊
Market Intelligence
Six technology market trends updated every 6 hours. Role demand catalogue refreshed every 2 hours with high/medium/low distribution.
🤝
Coaching Advice
Coaches submit a question about an employee (using their pseudo-ID only). AI responds with specific, actionable advice grounded in market context.
🧭
Skill Gap Analysis
AI maps the top 8 required skills for any target role, with required proficiency level and market demand score, cached for 12 hours.
📰
Intelligence Digest
A weekly digest synthesising connector-ingested market data into a structured report: top rising skills, role movements, and recommended actions.
ℹ️
Zero hallucination surface on user identity Every AI prompt uses anonymised profiles only. The system prompt for coaching advice explicitly says "Never reveal or speculate about personal identity or PII." Employee names, emails, and exact locations are never sent to any LLM.

AI Architecture

The AI layer is a dedicated microservice (ai-connector-service, port 8001) plus two Celery workers (celery-worker, celery-beat). All other backend services import directly from the shared RAG module — there is no inter-service HTTP call to the AI connector for inference; only for connector management (CRUD) and async task status polling.

# rag.py:120 — cache lookup with TTL check
async def _get_cached_response(cache_key: str, ttl_hours: int) -> list | dict | None:
    async with AsyncSessionLocal() as session:
        row = (await session.execute(
            sql_text("SELECT content, ingested_at FROM ai_documents "
                     "WHERE source_name = :key ORDER BY ingested_at DESC LIMIT 1"),
            {"key": cache_key}
        )).fetchone()

    if not row:
        return None
    gen_at = datetime.fromisoformat(row.ingested_at)
    if (datetime.now(UTC) - gen_at) > timedelta(hours=ttl_hours):
        return None              # expired — caller will regenerate
    return json.loads(row.content)
⚠️
Cache accumulation Old cache entries are never deleted — they simply age past their TTL and become invisible to the lookup query. The table grows over time. A scheduled cleanup task (not yet implemented) should prune entries older than 2× their TTL.

AI-Ops & Cost Resilience

A second cost & reliability layer sits in front of every conversational AI endpoint. The goal: shrink the input token bill on every provider, swallow duplicate requests for free, and stop a tripped provider from bleeding the whole replica fleet. Provider-specific optimisations (Anthropic prompt caching) sit behind a capability predicate so deployments on Perplexity / OpenAI / Ollama get the universal wins without any opt-in.

Five layers, in order

LayerWhat it doesEffect
1. Prompt-shape budget
grounded.py		 Top-3 opportunities (was 5), strategy block 800 chars (was 1500), chat history 4 turns (was 6). 25–35% input-token cut on every provider — universal first.
2. Response-hash cache
cache.py		 Redis key cf:airesp:<sha256>, 5-minute TTL, keyed on pseudo + question + workflow + skill_count + opp_top_ids. Double-clicks, page reloads, and accidental retries hit cache instead of the LLM. Cache hits are recorded as cf_suggestion_calls_total{outcome="cache_hit"}.
3. Multi-replica circuit breaker
quotas.py		 Per-provider open-state lives in Redis (cf:breaker:open_until:{provider}). After N consecutive failures the breaker trips for K seconds across all replicas. 3-replica deploy: one tripped replica protects the other two. Previously each replica had to re-hit the threshold independently.
4. Connector freshness gauge
metrics.py		 Unix-timestamp gauge cf_connector_last_sync_seconds{connector} set after each successful ingestion run. PrometheusRule alerts when time() − gauge > 86400 for 10m. Stale RAG context is no longer silently invisible.
5. Anthropic prompt cache
capabilities.py + rag.py		 Predicate supports_prompt_caching(provider); the Anthropic path sends system as [{type:text, cache_control:{type:ephemeral}}]. No-op for every other provider. ~90% input-cost cut on cache hits when deployed against Anthropic. Cache-creation/cache-read tokens recorded as cf_prompt_cache_tokens_total{provider,direction}.

Cache key construction

# services/ai_connector/cache.py
def make_cache_key(
    *, pseudo_id: str, question: str,
    workflow_id: str | None, skill_count: int,
    opp_top_ids: list[str], extra: list[str] | None = None,
) -> str:
    parts = [pseudo_id, question.strip().lower(), workflow_id or "",
             str(skill_count), ",".join(sorted(opp_top_ids))]
    if extra: parts.extend(extra)
    digest = hashlib.sha256("|".join(parts).encode()).hexdigest()
    return f"cf:airesp:{digest}"

/ai-chat additionally folds a hash of the recent conversation turns into extra= so a follow-up question with the same words doesn't collide with the original.

Provider-portability principle

Layers 1–4 ship benefits regardless of which AI provider the deployment uses. Layer 5 sits behind a capability predicate that returns False on Perplexity, OpenAI (which auto-caches server-side), Ollama, and Gemini. The same call site path will extend to Gemini's CachedContent by adding one entry to the predicate and a body-rewrite branch in _gemini_chat — no caller changes needed.

✓
Fail-open everywhere Cache, breaker, and quota helpers all degrade to no-op when Redis is unreachable. Better to serve traffic on a healthy LLM than to block the whole portal on a sidecar outage.

Daily quota visible to the user

Phase 1's daily-quota counter is enforced server-side, but historically the user couldn't see it — the first sign that you'd hit the limit was a 429 on your next AI call. GET /api/v1/quota/me exposes the same Redis counter the breaker enforces (no parallel state, no drift) and the employee portal sidebar polls it every 60s. The chip turns orange at ≤ 5 remaining so you get a heads-up before the ceiling.

Same fail-open posture as the rest of Phase 1: when Redis is down the chip just disappears (remaining = -1) rather than lying about state.

Drift gate (two-gate setup)

CI runs both gates in sequence, paired so neither bug class can slip through:

  • alembic check — fails when a SQLModel field has no matching migration column. The bug class that manifests as UndefinedColumn 500s after a fresh deploy. migrations/env.py uses an _include_object filter that skips index, FK, and unique-constraint naming divergence (cosmetic only) so this gate flags real drift cleanly.
  • pytest test_raw_sql_ghost_tables.py — fails when a router uses raw SQL against a table that has no SQLModel at all. The bug class behind the api_keys incident (table referenced via FROM api_keys, no model, no migration → 500 on first admin click). The alembic gate can't see those tables; this one tokenises every .py file in services/ and asserts each raw-SQL identifier maps to a real table or a tiny allowlist.

JSON Parsing with Graceful Fallback

# rag.py:81 — robust JSON extraction from LLM output
def _parse_json(text: str) -> list:
    if not text: return []
    text = text.strip()
    if "```" in text:                        # strip markdown code fences
        for part in text.split("```"):
            p = part.strip()
            if p.startswith("json"): p = p[4:].strip()
            if p.startswith("[") or p.startswith("{"): text = p; break
    try:
        result = json.loads(text)
        return result if isinstance(result, list) else [result]
    except json.JSONDecodeError:
        m = re.search(r"\[.*\]", text, re.DOTALL)  # last-chance regex
        if m:
            try: return json.loads(m.group())
            except: pass
    return []                                      # graceful empty on failure

Safety & Guardrails

Guardrails in CareerForge operate at three levels: prompt-level constraints, input validation, and infrastructure-level access control. The goal is to prevent PII leaks, hallucinated identities, runaway API spend, and prompt injection from malicious connector data.

GuardrailLayerImplementation
PII never sent to LLM Prompt Coaching prompts use pseudo_id + aggregated profile only. System prompt: "Never reveal or speculate about personal identity or PII."
4-pillar grounding (advice / chat / summary) Prompt The system prompt feeds only four named blocks — PROFILE, OPPORTUNITIES, INDUSTRY 4.0 READINESS, COMPANY STRATEGY — and explicitly forbids invented education, certifications, employer history, projects, salaries, locations, or metrics. Off-context questions get a clean refusal asking the user to add the missing detail to their profile.
Skill UUIDs resolved to names Prompt Skill IDs are joined against SkillTaxonomy before composing the prompt; the model never sees raw UUIDs. Custom user-typed skills round-trip as their name.
Citation & UUID scrub Post-processing Output passes through a regex pass that strips Perplexity-style citation markers ([1][2], [digest], [cache_roles]) and any stray UUIDs that survived the prompt rules. Whitespace and orphaned punctuation are normalised.
Profile-completeness gate API AI Advice / Career Summary / Career Advisor refuse with 422 when the profile is too thin (specialisation + seniority + 3 skills + a career goal). Better a clear "complete your profile first" than a confident fabrication.
Auditable suggestion trail DB Every AI surface (role suggestions, learning path, skill gap, profile summary, AI advice, AI chat, mentor match, opportunity match, market trends, emerging roles) writes an immutable row to ai_suggestion_logs with the suggestion type, employee pseudo-id, model used, and a context_used breakdown. Surfaced in the admin Suggestion Audit page with summaries that lead on the relevant identity.
Outcome feedback (accepted / dismissed / rated) DB + API Each suggestion log can be paired with one or more ai_suggestion_feedback rows. Employee endpoint POST /ai-suggestions/{log_id}/feedback validates ownership; admin endpoint /audit/feedback-summary aggregates acceptance rate and average rating per suggestion_type. The cf_suggestion_calls_total{outcome="feedback_accepted|feedback_dismissed|feedback_rated"} counter makes the same signal visible in Grafana.
Per-employee daily quota Redis Default 50 LLM-backed AI calls per pseudo_id per UTC day. /ai-advice and /ai-chat consume one slot each; quota exhausted → HTTP 429 with a clear message; resets at 00:00 UTC. Backed by Redis INCR + EXPIRE; degrades open if Redis is unreachable.
Per-provider circuit breaker Process state After 5 consecutive failures (http_error / exception) the breaker opens for 120s. Open-state short-circuits the LLM call and returns "" so callers serve the configured fallback message instead of a 5xx. Auto-closes after the cooldown. Visible at GET /ai-workflow/cost-summary and indirectly via the cf_suggestion_calls_total{outcome="blocked_breaker"} counter.
Monthly token cost cap Redis + Workflow Config Token estimate (~chars/4) recorded per provider in cf:cost:YYYY-MM Redis hash. Set coverage_config.__cost__.monthly_token_cap on the active workflow; once hit, every LLM call returns "" until the next month. Unset = unlimited. Surfaces as { providers: { perplexity: { in, out }}, total } in the admin cost-summary endpoint.
Audit evidence trail (verbatim prompt + answer) DB + API payload_json on conversational suggestions stores the full system prompt, the full user prompt (the four grounding blocks verbatim), and the full answer. Admin endpoint GET /audit/suggestions/{log_id}/evidence returns a downloadable text/markdown document with system prompt, user prompt, answer, context-used breakdown and every feedback event — ready to print-to-PDF for compliance / HR review.
JSON-only output (cataloguing endpoints) Prompt For listing endpoints (role suggestions, market trends, emerging roles), the system prompt ends with "Respond ONLY with a valid JSON array". _parse_json() strips markdown fences and falls back to empty list. Conversational endpoints (advice / chat / summary) return Markdown rendered safely via the shared CfMdText component (marked + DOMPurify).
Input length cap API Coach advice question: max 2000 chars (FastAPI validator, advice.py:42). 422 returned if exceeded.
LLM timeout HTTP httpx.AsyncClient(timeout=45.0) for all LLM calls. Connector fetches use 60s timeout. Prevents stuck requests blocking workers.
Celery task isolation Infra task_acks_late=True, worker_prefetch_multiplier=1: tasks only ACK on success, preventing duplicate processing on crash.
Connector dedup Ingestion SHA-256 content hash checked before insertion. Prevents prompt poisoning from re-injecting modified documents at the same URL.
Encrypted connector config DB Connector API keys stored in config_json_encrypted (AES-256-GCM via Vault-backed key). Never logged.
API key injection Infra LLM keys come from Kubernetes Sealed Secrets → env vars. Never in code or ConfigMaps.
Retry cap Celery max_retries=3 on connector sync tasks. Failed ingestion writes to IngestionHistory.error_message and stops retrying after 3 attempts.
CORS allow-list Network Only the four portal domains are allowed; wildcard CORS is explicitly not configured.

PII Protection

Personal data is protected at two levels: pseudonymisation of the employee identity before it reaches any AI prompt, and AES-256-GCM encryption of the stored PII fields.

Pseudonymisation Flow

Keycloak sub
(real user ID)
→
HMAC-SHA256
with HMAC_KEY_HEX
→
pseudo_id
(opaque token)
→
Sent to LLM
(no name/email)

Coaches see only the pseudo_id, the employee's specialisation, seniority, experience years, and region (not city). The HMAC key is injected from Sealed Secrets — not derivable from any data in the LLM prompt.

At-rest Encryption

Employee PII fields (name, email, exact location) are stored encrypted in employee.encrypted_pii using AES-256-GCM. The key is injected at runtime from DEV_PII_KEY_HEX (Vault-backed in production, Sealed Secret in the demo cluster).

Strengths & Weaknesses

An honest assessment of what the current AI implementation does well and where it falls short.

Strengths

✓
Dual-LLM fallback with graceful degradation The Claude → Perplexity → empty chain means the platform stays usable even if the primary LLM is down. Users see a "being configured" message rather than a crash.
✓
AI-surface observability Prometheus counters & histograms cover the bits the HTTP-level instrumentator can’t see: cf_llm_calls_total (per provider/model, with outcome success | empty | http_error | exception), cf_llm_call_duration_seconds histogram (0.5…90s buckets), cf_suggestion_calls_total per suggestion_type & portal (incl. refused_thin_profile / empty_answer / feedback_*), and cf_llm_artifacts_scrubbed_total — if Perplexity citations or UUIDs start leaking again, that counter spikes and the alert fires before any user notices.
✓
Schema/migration drift gate (CI) CI runs alembic check after alembic upgrade head; a SQLModel class without a matching migration fails the build. A pytest under tests/integration/ runs the same check locally. Closes the class of bug where production 500-ed with UndefinedTableError on the first call to a new endpoint.
✓
Cost & abuse controls Three layers, all observable in Grafana: a per-employee daily quota (default 50, configurable, Redis-backed, UTC reset), a per-provider circuit breaker that opens after 5 consecutive failures and short-circuits LLM calls for 120s, and a monthly token cost cap configurable per active workflow (coverage_config.__cost__.monthly_token_cap). Admin endpoint /ai-workflow/cost-summary returns the live state. Fail-open on Redis outage so a sidecar problem doesn’t take the AI surface down.
✓

Weaknesses

✗
No automatic outcome-feedback events The ai_suggestion_feedback table accepts the auto action but nothing emits those events yet. Without them, signal stays dependent on humans clicking thumbs-up / dismiss — rare enough that the rules engine on top can’t fire reliably. Wait for manual feedback volume before building.
✗
No structured-output value-range validation Conversational surfaces are hardened against fabrication (4-pillar grounding, anti-citation system prompt, post-hoc UUID/citation scrub, 422 refusal on thin profiles). ✗
Single embedding dimension assumption The schema hardcodes Vector(1536). Switching to a newer embedding model with different dimensions requires a schema migration.

  • Coach caseload variation — some coaches cover two practices; their AI advice calls span different specialisations, stress-testing the system prompt's generalism
  • Temporal dynamics — caches expire at different rates; the snapshot of "current" AI output shifts over the 2h/6h/24h/7d cycle even with no user action

  • Practice-level cohort patterns — employees in the same practice tend to receive similar role suggestions (same specialisation cache key), surfacing whether the LLM's recommendations are varied enough

Reconciliation Loops

A reconciliation loop is a closed feedback cycle: the platform observes the outcome of its AI recommendations, compares them to what actually happened, and adjusts future recommendations accordingly. CareerForge has the infrastructure to support several such loops — some partially built, some requiring additional work.

Loop 1 — Connector Quality Feedback

Current state: fully implemented infrastructure, loop not closed.

Every connector source has a trend_score field and last_status_code. The ingestion task records IngestionHistory.chunks_ingested and success. A reconciliation job could:

  1. Correlate connector sources with AI output quality (do documents from source X lead to higher-confidence market trends?)
  2. Automatically lower trend_score for sources that repeatedly 404 or return empty content
  3. Promote sources whose topics appear frequently in accepted learning paths

Loop 2 — Learning Path Acceptance Rate

Current state: data exists, loop not implemented.

The learning_path_nodes table records which nodes employees mark complete. A weekly digest job could compute:

-- Proposed reconciliation query
SELECT node_type, skill_name,
       COUNT(*) FILTER (WHERE status = 'completed') AS completed,
       COUNT(*) FILTER (WHERE status = 'skipped')  AS skipped,
       COUNT(*) AS total
FROM learning_path_nodes
WHERE created_at > NOW() - INTERVAL '30 days'
GROUP BY node_type, skill_name
ORDER BY (skipped::float / NULLIF(total, 0)) DESC;

Nodes with high skip rates could feed back into the system prompt: "Avoid recommending X for specialisation at seniority level — historically low completion rate."

Loop 3 — Role Suggestion Relevance

Current state: data partially exists.

If an employee clicks through a role suggestion and sets it as their target role, that is an acceptance signal. The ratio of suggested-to-accepted roles, broken down by specialisation and seniority, tells us whether the LLM is recommending roles people actually want. This can be used to:

  • Adjust the role suggestion prompt to reflect which roles in which practices actually get accepted
  • Build a fine-tuning dataset of (profile → accepted role) pairs
  • Invalidate the 24h cache early for a sub-segment if the acceptance rate drops below a threshold

Loop 4 — Coaching Advice Quality Score

Current state: not implemented. Requires UI addition.

Add a thumbs-up/down widget after each AI coaching response. Store the score in a new ai_feedback table keyed by (pseudo_id, question_hash, model_used, score). The reconciliation job then:

  1. Computes per-model quality scores over rolling 30-day windows
  2. Routes to the higher-scoring model when both are available
  3. Feeds low-scored (question, answer) pairs back to the prompt as negative examples

Loop 5 — Market Trend Accuracy

Current state: infrastructure exists for trend_direction tracking. Loop not closed.

Perplexity sonar-pro returns real-time web search context. Store the predicted trend_direction at time T. At time T+30d, re-run the trend query and compare. If the direction is consistent, increase the confidence score for that topic. If it reversed, lower it and flag for human review. Over time this builds a calibration curve for the AI's trend forecasting ability.
