What AI Does in BankOffer

AI in BankOffer is not a chatbot — it is the core decision engine. It classifies customers, scores products for relevance, ranks offers, generates market intelligence, and proposes new products for the catalog. Every AI call produces structured JSON output, and every action is logged to an immutable audit trail.

🎯
Offer Personalization
Claude scores each product against the customer's life stage, risk tolerance, income, and spending patterns. Top 5 ranked offers, no duplicates.
👤
Customer Profiling
Rule-based life stage classification (5 stages) and weighted risk scoring (1–10) from 5 demographic factors. Segment derivation: high_net_worth, strong_saver, leveraged.
📊
Market Intelligence
5 intelligence categories (Exchange Markets, Geopolitics, Regulations, Economic Indicators, Trends), 16 real-time signals scored for impact and relevance.
💡
AI Product Suggestions
AI generates new product ideas with confidence scores, target segments, and market drivers. Admin approval workflow with 24h staging cool-down before going live.
🛡️
Compliance Automation
MiFID II suitability hard-filter blocks mismatched products before scoring. GDPR Art. 22 automated-decision disclosures generated per offer batch.
📋
Audit & Transparency
Every AI call logged to ai_api_call_log. Every offer batch snapshotted in audit_recommendations. Public transparency endpoint per AI Act Art. 13.
ℹ️
Zero PII sent to LLMs Customer emails are stored only as irreversible SHA-256 hashes. Internal customer IDs are replaced with pseudonymized UUIDs before any AI call. Personal data never leaves the platform's boundary.

System Architecture

Three microservices (API, Worker, Notification) backed by PostgreSQL and Redis. The Worker runs the profiler, scorer, and ranker as Python modules — no inter-service HTTP call for inference. The API handles connectors, intelligence, and offer delivery. Deployed on Kubernetes via ArgoCD GitOps.

Customer Profiler

The profiler runs entirely on deterministic rules — no ML model, no LLM call. It outputs a ProfileResult with life stage, risk score (1–10), and derived segments. Results feed directly into the scorer's context.

Life Stage Classification

Priority-ordered rule evaluation. The first matching rule wins; falls back to mid_career.

StageConditions
retiredage ≥ 67
pre_retirement55 ≤ age < 67
young_family25 ≤ age ≤ 40 AND dependents ≥ 1
new_graduateage < 25 AND account_tenure ≤ 2 years
mid_careerage 35–55 AND income ≥ 60 000 · Default fallback

Risk Tolerance Scoring — Weighted Formula

Five factors combined with fixed weights into a 1.0–10.0 score (higher = more risk tolerant):

age factor (25%)
younger → more tolerance
income factor (15%)
log-scaled
investment ratio (25%)
inv / income
savings cushion (15%)
savings_ratio
leverage penalty (20%)
loan / income → penalty
# services/worker/profiler.py:59 — risk tolerance scoring
def score_risk_tolerance(age, annual_income, investment_balance, savings_ratio, loan_to_income):
    age_factor    = max(1.0, min(10.0, 10.0 - (age - 18) * (7.0 / 62.0)))
    income_factor = min(10.0, 1.0 + math.log10(max(1, annual_income / 10000)) * 3.0)
    inv_factor    = min(10.0, 1.0 + (investment_balance / annual_income) * 3.0)
    savings_factor = 1.0 + savings_ratio * 9.0
    leverage_penalty = min(5.0, loan_to_income * 5.0)

    return round(max(1.0, min(10.0,
        0.25 * age_factor + 0.15 * income_factor + 0.25 * inv_factor
        + 0.15 * savings_factor - 0.20 * leverage_penalty
    )), 2)

Product Scorer

The scorer calls Claude once per product. Each call is a focused prompt asking for a relevance_score, confidence_score, and a one-sentence personalization_reason. Failed calls produce a 0.0/0.0 default — the product is included but will be filtered by the ranker's threshold.

System Prompt

# services/worker/scorer.py:50 — SCORING_SYSTEM_PROMPT
"""You are a bank product recommendation engine. Given a customer profile and a product,
output a JSON object with exactly these fields:
  - relevance_score: float 0.0-1.0 (how relevant this product is to the customer)
  - confidence_score: float 0.0-1.0 (how confident you are in this assessment)
  - personalization_reason: string (one sentence explaining the recommendation)

Base your scores on the customer's life stage, risk tolerance, income, segments,
and spending patterns. Only output valid JSON. No markdown fences."""

Prompt Assembly

# services/worker/scorer.py:60 — per-product prompt
f"Customer: id={context.customer_id}, life_stage={context.life_stage}, "
f"risk_score={context.risk_score}, segments={context.segments}, "
f"income={context.annual_income}, top_spending={context.spending_categories[:5]}\n\n"
f"Product: id={product['id']}, name={product['name']}, type={product['type']}\n\n"
"Score this product for this customer."
ℹ️
Scores are clamped server-side Even if the LLM returns a value outside 0–1, the scorer clamps it: max(0.0, min(1.0, raw)). JSON extraction falls back to regex for responses wrapped in markdown fences.

LLM Protocol (injectable)

The scorer depends on the LLMClient Protocol — any class with create_message(prompt, system, max_tokens) → str works. This makes unit testing trivial and provider swaps zero-cost.

Offer Ranker

The ranker takes the full list of scored products and applies four sequential filters before returning the final offer list. This is where business rules, consumer-protection cooldowns, and diversity constraints are enforced — entirely without another LLM call.

Five-Step Ranking Pipeline

1. Threshold Filter
→
Drop relevance < 0.15 or confidence < 0.10
2. Cooldown Filter
→
Skip products dismissed within the last 30 days
3. Diversity Constraint
→
Max 2 products of the same type (savings, credit_card…)
4. Truncate
→
Top 5 by relevance_score
5. Assign Ranks
→
Generate UUID offer_id, assign rank 1–5

Configurable at runtime All thresholds are overridable via RankingConfig. A/B testing can swap configs without code changes — and the ranker logs counts at each filter step via logger.debug for Grafana visibility.

10 AI Guardrails

All guardrails are enforced at the API layer in services/api/routers/intelligence.py. They run before any LLM call is made and on every AI-generated output before it is processed.

#GuardrailMechanism
01Confidence thresholdSuggestions with confidence < 0.60 are rejected and logged. Prometheus counter AI_SUGGESTIONS_FILTERED.
02Rate limitingMax 5 /analyze calls per hour per instance. Redis-backed counter. HTTP 429 on breach. Prometheus counter AI_ANALYSIS_RATE_LIMITED.
03PII content filterRegex strips email, phone, SSN, card numbers, and Romanian IBANs from every AI output before storage or display.
04Prompt injection detectionPattern match for "ignore previous", "you are now", <|im_start|>, etc. Matched text replaced with [FILTERED], warning logged.
05Product catalog capMax 50 active products. AI-suggested products blocked from publishing if catalog is at capacity.
06Dual approvalThe admin who implements a product cannot be the same who approved it. Enforced at DB level.
0724-hour stagingCOOLDOWN_HOURS = 24 — AI products cannot go live immediately after approval.
08Auto-expiryPending suggestions older than 30 days are automatically expired. Prevents stale AI decisions from sitting in the queue.
09Fairness monitoringPrometheus histograms track offer acceptance rates by age bracket and income tier. Grafana alert if disparity exceeds threshold.
10Audit immutabilityPostgreSQL BEFORE UPDATE OR DELETE triggers on all 4 audit tables. HTTP 500 if audit write fails — no offer without a record.

EU Compliance

Five EU regulatory frameworks apply simultaneously to a banking AI system. Every feature in BankOffer AI was built compliance-first. The Consent Registry syncs from an Excel workbook (5 sheets) with SHA-256 change detection and auto-versioning.

RegulationRequirementImplementation
GDPR
Art. 5, 6, 7, 22, 30		 Consent, pseudonymization, auto-anonymization, automated decision rights, processing records 5-tier consent toggles · SHA-256 email pseudonymization · 2-year auto-anonymization · Art. 22 disclosure per offer batch · audit_log as Art. 30 record
EU AI Act
Art. 9, 12, 13, 14		 Risk register, logging, transparency disclosures, human oversight 10-entry risk register · ai_api_call_log (full prompt + response) · Public transparency endpoint · Kill switch with full audit trail · Algorithmic explanation panels
MiFID II Product suitability, risk profile matching, excluded-products disclosure Hard suitability filter before scoring (not a soft recommendation) · Per-product risk profile matching · Excluded products list with reasons shown to customer
EBA Guidelines Fair treatment, product governance, distribution controls Fairness Prometheus histograms by age/income · Product catalog governance workflow · Distribution consent per channel
ePrivacy Directive Per-channel marketing consent Separate consent flags for push, email, SMS · Notification service checks flags before delivery · Consent history append-only log
⚠️
Kill Switch (AI Act Art. 14) The kill switch halts all AI offer generation platform-wide. It writes to the audit_log table (immutable), broadcasts a notification to all staff, and causes the offer endpoint to return a compliance-safe empty response until re-enabled by an authorized admin.

Audit System

Four immutable PostgreSQL tables. Database-level BEFORE UPDATE OR DELETE triggers prevent any modification or deletion. The offer engine returns HTTP 500 if the audit write fails — recommendations cannot be served without a record.

TableWhat It CapturesRegulation
audit_log Every state-changing action: actor, action, resource, before/after values, IP, user-agent, endpoint, HTTP method/status, duration, correlation request_id GDPR Art. 30
ai_api_call_log Every LLM interaction: request prompt, response text, model, provider, tokens used, latency ms, HTTP status, guardrail outcome, errors AI Act Art. 12
consent_history Append-only record of every consent flag change: old value → new value, who changed it, timestamp, channel GDPR Art. 7(1)
audit_recommendations Full snapshot per offer batch: input features, profile result, suitability check outcomes, scored products, exclusions, consent state, model version used AI Act Art. 12
🔒
Database-level immutability PostgreSQL BEFORE UPDATE OR DELETE triggers are installed on all four tables. Even a direct psql connection by a DBA cannot modify or delete a row — the trigger raises an exception. The only permitted operation is INSERT.

Privacy by Design

Privacy controls are structural — they are embedded in the data model, not bolted on as policies. No personal data is stored in a reversible form, and no PII ever reaches an LLM.

ControlHow It Works
Email pseudonymizationSHA-256 hash stored; plaintext email discarded at registration. Hash is used for deduplication only.
Customer ID isolationDatabase internal IDs (51, 52…) replaced with pseudonymized UUIDs for all external API responses.
Auto-anonymizationProfiles older than 2 years are automatically stripped of quasi-identifiers by a scheduled background task.
LLM context sanitizationScorer and intelligence prompts contain only pseudonymized IDs, life stage, risk score, segment labels, and anonymized spending categories — never name, email, or address.
Consent-gated AIProfiling and automated decision consent flags are checked before any AI inference. No consent = no AI call = no personalized offers.

Strengths & Weaknesses

Strengths

✓
Compliance-first architecture Five EU regulations handled simultaneously in the data model. No post-hoc compliance patches needed.
✓
Real code, real guardrails 10 guardrails are enforced in production code (not documentation). PII filter, rate limiter, confidence threshold, and prompt injection detection all run on every AI call.
✓
Injectable LLM protocol The scorer depends on a Protocol, not a concrete client. Any provider — Claude, GPT, Gemini, or a mock — slots in identically. Testing needs no mocking infrastructure.
✓
Deterministic profiler Life stage and risk scoring are pure functions — no ML model to retrain, no drift, fully unit-testable. Edge cases are auditable by reading 60 lines of code.
✓
Audit immutability at DB level PostgreSQL triggers enforce write-once semantics. No application-layer policy can circumvent it.

Weaknesses

✗
One LLM call per product Scoring N products makes N serial LLM calls. With 8 products that is 8 × latency. No batching or concurrent scoring implemented yet.
✗
No scoring cache The same customer profile + product pair is re-scored on every request. Identical inputs produce a fresh LLM call with potentially different outputs (non-determinism).
✗
Profiler rule gaps The classify_life_stage function has a dead branch (line 50: 25 ≤ age < 25 is always false). Edge cases between stages can produce counterintuitive results.
✗
Rate limiter is per-instance The Redis rate limit counter is keyed per API instance, not per customer. In a multi-replica deployment it is effectively 5 × N calls/hour.
✗
No ML model in production ml/ has training scaffolding but the inference path is not wired into the offer pipeline. Relevance scoring depends entirely on the LLM, with no offline-trained fallback.
